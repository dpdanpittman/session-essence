# Reviewer Report — P-session-essence-audit / perf

VERDICT: Request Changes

This is a small surface area (one `~340-line` Node file, one ~90-line shell hook) so most performance findings concern _correctness under degraded conditions_ rather than algorithmic hot paths. The single synthesis pass through `chat()` is unambiguously the hot path — three sequential remote calls with no caching, no retry, and no per-call timeout on the Claude branch. The Ollama branch has a timeout but the timeout strategy is brittle: a hard cap that loses the prior two passes' work when it fires. The PreCompact script has a check-then-touch lock that is non-atomic and a per-second archive filename that can collide. None of these will explode on Dan's laptop today, but several will fire the first time the system is run against a long-running session or by an operator with a busier hook config. Three of these are warning-grade because they intersect with intent-doc claim 7 ("safe under concurrent invocation"), claim 4 ("truncation is bounded" — bounded by line _count_, not byte volume), and claim 5 (lazy init — the cost story is fine but the failure-mode story isn't).

The most important single defect is **the absence of a per-call timeout or retry on `claudeChat()`**. Anthropic's SDK has its own default timeouts but they are minutes-scale, and a hung first or second pass means an MCP tool invocation that just sits there from the client's perspective with no signal to the user. Combined with the fact that `synthesize_essence` is documented as a tool the calling Claude invokes inline (not via the detached `claude -p` script), this is a tool-response budget problem waiting to happen.

The response-budget question (claim from plan §3, "Token cost on Claude backend") works out comfortably for the _output_ path: psych (3000) + socio (3000) + merge (2000) caps response at ~8K tokens / ~31KB of text — well inside any reasonable MCP per-response budget. The _input_ path is the actual exposure: 200 lines × ~2KB/line × 3 sequential calls = ~350K input tokens worst case (~$0.39 at Haiku pricing per invocation in the pathological case). That's not catastrophic but it's also not the "few cents per run" that the README claims, and an operator who turns on the PreCompact hook expecting cheap synthesis can find their bill nonlinear in session length without it ever being visible in the tool response.

## Findings

### F-PERF-001 — Hard 10-minute Ollama timeout discards two completed passes mid-merge

**Severity**: warning
**Citations**: `index.js:37`, `index.js:94-103`, `index.js:197-212`
**Scenario**: Operator running Ollama backend on hardware where `qwq:32b` averages ~4 minutes per pass (a 24GB-VRAM card with some CPU offloading — README's own troubleshooting section acknowledges this case at line 308). Psychologist completes in 3:45; sociologist completes in 4:10; merge call hits `OLLAMA_TIMEOUT=600000` (10 min total per `fetch` call, but each call has its own clock so this is per-call, not cumulative — actually fine for individual calls, but the _cumulative_ synthesis run can exceed 12 minutes wall-clock with no checkpointing). If any single pass exceeds 10 minutes, `AbortController.abort()` fires and the whole tool invocation throws — the prior pass's content is lost. There is no resume, no partial-merge fallback.
**Reproduction**: set `OLLAMA_TIMEOUT=5000` (5s) and call `synthesize_essence` against Ollama with a normal observation set. The first call aborts, the tool returns `Synthesis error: ...`, and the two-or-three minutes the model just spent generating are unrecoverable.
**Suggested defense**: (a) cache `psychResult` and `socioResult` in-memory keyed by a content hash of `(systemPrompt, logText)` so a retry of the same `synthesize_essence` call doesn't re-pay for completed passes; (b) optionally short-circuit the merge to a minimal "psych-only" or "socio-only" portrait when the merge call fails — a degraded portrait is better than none; (c) document the 10-minute-_per-call_ (not per-run) semantics in README — the table at line 290 implies cumulative, which is wrong.
**Affects intent claim**: claim 4 (truncation bounded) — the bound is fine but the cost of _paying_ the bound 3× is what blows the budget.

### F-PERF-002 — No timeout or retry on `claudeChat()`; transient 429s / 529s propagate as MCP tool errors

**Severity**: warning
**Citations**: `index.js:57-77` (no timeout, no retry wrapper), `index.js:126-139` (dispatcher), plan §3 ("No backoff or retry. If the Anthropic API is rate-limited or transiently fails, what happens? `claudeChat()` propagates the error to the tool response.")
**Scenario**: Anthropic's API rate-limits the account (HTTP 429) or returns an overloaded-error (HTTP 529 — common around US business-hours peaks). `client.messages.create()` rejects, the rejection propagates through `chat()` → the `catch (err)` at `index.js:245` returns `Synthesis error: <message>` to the MCP client. The calling Claude sees a one-shot failure, no auto-retry. Worst case: a `synthesize_essence` call that hit the third (merge) pass after two minutes of successful work fails on a 60-second transient and the operator has no signal that "this was retryable, try again in 30 seconds." The Anthropic SDK has built-in retries with exponential backoff (default 2 retries per call) so this is partially mitigated _per call_, but cross-call resume after a final-retry-exhausted failure is still absent.
**Reproduction**: temporarily set `ANTHROPIC_API_KEY=invalid_key_xxx`, call `synthesize_essence` — the error message surfaces the SDK error verbatim. Now consider that with a valid key but rate-limited account, the same path runs, then fails after the SDK's default backoff is exhausted, with no caching of the prior two pass results.
**Suggested defense**: same in-memory cache as F-PERF-001 (key by `(content_hash, model_name)`); explicitly set a `timeout` option on `client.messages.create` (Anthropic SDK supports `timeout` per-request) — recommend ~120s per pass, ~360s total wall-clock budget; surface a structured `retryable: true|false` flag in the tool error response so the calling Claude can decide whether to retry.

### F-PERF-003 — Lock-file race in PreCompact script is check-then-touch, not atomic

**Severity**: warning
**Citations**: `examples/synthesize-essence.sh:17-29`, intent claim 7 ("safe under concurrent invocation")
**Scenario**: Two PreCompact-triggering events fire within the same ~10–20ms window (e.g., manual `/compact` + automatic context-fill compaction in the same Claude Code session, or — more realistically — two Claude Code sessions sharing `~/.claude/essence/` each receiving compaction signals near-simultaneously). Process A runs `test -f $LOCK_FILE` (false), proceeds to `wc -l`, passes the `>= 15` check. Process B does the same. A `touch`es the lock, B `touch`es the lock (idempotent). Both spawn detached `claude -p` instances. Both race on the same `portrait.md` (Read + Write tool); both archive `observations.jsonl` to a filename derived from `date +%Y%m%d-%H%M%S` — and per-second precision means the two archive filenames _collide_ if both `cp`s land in the same wall-clock second. Both clear `observations.jsonl`. The slower archive `cp` may read a now-truncated source (race between A's clear and B's cp). Race window is small (estimate 10–20ms between the test-`f` and the `touch`) but the consequence is dropped observations, two `claude -p` Bash sessions fighting over the same portrait, and a `last-synthesis.log` whose contents reflect an interleaved write.
**Reproduction**: in a shell, `bash examples/synthesize-essence.sh &; bash examples/synthesize-essence.sh &; wait` against a populated `~/.claude/essence/observations.jsonl`. Inspect `~/.claude/essence/archive/` — there will be one or zero files when there should be two; both `claude -p` processes ran (`pgrep claude | wc -l`); `~/.claude/essence/observations.jsonl` is empty.
**Suggested defense**: replace check-then-touch with atomic acquisition: `if ! ( set -o noclobber; > "$LOCK_FILE" ) 2>/dev/null; then echo "already running"; exit 0; fi` — `noclobber` makes the redirect fail if the file exists, atomically. Also add `mkdir`-based fallback: `mkdir "$LOCK_FILE.d" 2>/dev/null || { echo "already running"; exit 0; }` works on every POSIX shell. Second fix: extend archive timestamp to millisecond precision (`date +%Y%m%d-%H%M%S-%N | head -c 22`) so concurrent archives don't collide on filename even if the lock fails.
**Affects intent claim**: claim 7 (directly invalidated as currently written).

### F-PERF-004 — `synthesize_essence` response holds entire psych + socio + merge text; predictable 25–35KB per call

**Severity**: suggestion
**Citations**: `index.js:220-243` (output construction includes full psych + socio + merge inline)
**Scenario**: Plan §3 raised concern about MCP per-tool-response budget. Measurement: `max_tokens` ceiling is 3000 + 3000 + 2000 = 8000 output tokens, ~30KB of text. Plus boilerplate (headers, `<details>` wrapper, attribution footer) ~500 bytes. Total response payload ≈ 31KB worst case. This is well within MCP transport limits (default ~25MB), within Claude Code per-tool-response budget (no published cap but observed tools regularly return >100KB without truncation). NOT a critical concern. However: the Analysis Details `<details>` block contains the full psych and socio reports — material the calling Claude rarely needs (intent claim 10 says it's appendix, collapsed by default). On token-counted MCP transports (some clients bill MCP tool output against the model's context window), this is ~6000 tokens of always-attached appendix that the model may or may not benefit from.
**Reproduction**: `node index.js`, drive it with a 200-line observations input, capture the response. Measure `len(response.content[0].text)` ≈ 31KB; psych+socio reports account for ~24KB / ~6000 tokens of that.
**Suggested defense**: add an optional `include_observer_reports: boolean` parameter to `synthesize_essence`, defaulting to `true` to preserve current behavior, but allowing callers (especially the PreCompact script if it ever migrated to MCP) to request only the merged portrait. Saves ~75% of response token count for the common-case "just give me the portrait" caller.

### F-PERF-005 — `wc -l` on `observations.jsonl` is O(N) per PreCompact firing; archive accidentally piped through has unbounded effect

**Severity**: suggestion
**Citations**: `examples/synthesize-essence.sh:22`, plan §3
**Scenario**: Normal `observations.jsonl` is ≤500 lines (PreCompact clears it every cycle). `wc -l` on 500 lines is <1ms — total non-issue. The failure mode is when `OBS_FILE` is misconfigured (operator points the env at `~/.claude/essence/archive/all.jsonl` after a tidy-up) or the PreCompact script crashes mid-flight without clearing observations — file grows monotonically. At N=100,000 lines (a year of un-cleared observations at 250/day) `wc -l` is still ~3–5ms wall-clock — _still_ not actually a problem. So this is purely a hygiene finding: the script reads the whole file just to count. `awk 'END{print NR; exit}'` is equivalent; `tail -n +1 $OBS_FILE | head -c 1` proves non-emptiness for ~0 cost; but honestly the existing `wc -l` is fine for any realistic operator state.
**Suggested defense**: leave as-is. Document in README that `observations.jsonl` should be pruned (it already auto-clears on every PreCompact, so this is more about the failure-mode recovery path: "if synthesis fails to clear it, manually archive + truncate"). Optional: replace `wc -l` with `awk 'NR>=15{exit 0} END{exit 1}'` which short-circuits at line 15 — O(1) when there are enough observations.

### F-PERF-006 — Polling loop with 5s `sleep` to clean up lock file is fragile to script exit before cleanup spawns

**Severity**: suggestion
**Citations**: `examples/synthesize-essence.sh:89-92`
**Scenario**: Line 89 captures `CLAUDE_PID=$!`. Line 90 spawns a _second_ background subshell that polls `kill -0 $CLAUDE_PID` every 5 seconds until the `claude -p` process exits, then `rm -f "$LOCK_FILE"`. Line 92 echoes "synthesis launched" and the parent script exits. Failure modes: (a) if the parent script is killed (SIGKILL) between lines 89 and 90, the cleanup subshell never spawns and the lock file persists forever, requiring manual `rm`. (b) if the system reboots while `claude -p` is running, both the parent script and the cleanup subshell die; next session sees a stale lock. (c) the 5s polling interval means up to 5 seconds of lag between `claude -p` exit and lock release — small, but if synthesis runs are short (~30s) that's a 16% extra hold window during which a _legitimate_ next PreCompact will refuse to fire.
**Reproduction**: kill the parent script with SIGKILL immediately after launching; observe `.synthesis-running` left behind permanently.
**Suggested defense**: stale-lock recovery via PID-in-lockfile: `echo $$ > "$LOCK_FILE"` instead of `touch`, then on next run, `if [ -f "$LOCK_FILE" ] && ! kill -0 "$(cat $LOCK_FILE)" 2>/dev/null; then rm "$LOCK_FILE"; fi` before the lock test. Alternatively use `flock -n` for true file-lock semantics that release on process exit regardless of signal disposition: `flock -n "$LOCK_FILE" -c "synthesis_body" || echo "already running"`.

### F-PERF-007 — Token accounting in synthesis response is correct for Claude but loses metadata on Ollama

**Severity**: suggestion
**Citations**: `index.js:214-218` (token tally construction), `index.js:111-116` (Ollama response shape — exposes `eval_count` but not input/prompt eval count separately)
**Scenario**: Operator switches `SYNTHESIS_BACKEND` between calls and tries to compare costs. Claude branch reports `Input: X / Output: Y` (accurate, sums across 3 passes). Ollama branch reports `Tokens: X` where X is `eval_count` summed across 3 passes — but `eval_count` in Ollama's API is _output_ tokens only; `prompt_eval_count` (input) is silently dropped. The two backends report apples-to-oranges, and there's no wallclock figure at all (despite Ollama exposing `total_duration` at `index.js:114`).
**Reproduction**: run a synthesis on each backend with the same observation log; compare the attribution footer in `# Session Essence Portrait`. Ollama reports look ~3× smaller than Claude reports for equivalent work.
**Suggested defense**: on the Ollama branch, sum `data.prompt_eval_count` + `data.eval_count` and report as `Input/Output` to match the Claude shape; add a `Wallclock: Ns` field by summing `total_duration / 1e9` from each Ollama call (and `Date.now()` deltas on the Claude branch).

## Cross-reviewer notes

**For arch lens**: F-PERF-004 raises a question about whether the merged portrait + observer reports really belong in the same response, or whether the appendix should be a second tool call (`get_observer_reports`) gated on the synthesis having just run. That's an architecture concern (tool surface design) more than a perf concern.

**For sec lens**: F-PERF-003's race-condition fix (PID-in-lockfile + `flock`) reduces a TOCTOU exposure that the security lens should weigh independently — if a hostile process can write to `~/.claude/essence/` it can poison the lock contents. Also, F-PERF-002's suggestion to surface `retryable: true|false` in the error response intersects with whether transient-failure messages leak account state (rate-limit vs. invalid-key are differently sensitive). Both are primarily perf findings here; surfacing for cross-validation.

**For both**: the `nohup claude -p ... --permission-mode bypassPermissions` invocation on `examples/synthesize-essence.sh:80-86` is a perf-irrelevant but worth-noting line — the bypass-permissions stance crosses into security territory and the long-running detached process with file-write capability against `$HOME` crosses into both arch (where does this kind of orchestration belong?) and security (what stops the detached process from running off the rails?). I'm not filing it as a perf finding but it's the most operationally consequential line in the script.
