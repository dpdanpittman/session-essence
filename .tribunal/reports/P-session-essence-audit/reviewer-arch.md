# Reviewer Report — P-session-essence-audit / arch

VERDICT: Request Changes

The codebase is small, cohesive, and the architectural intent is clearly articulated in `docs/design.md` and `AGENTS.md`. The single-file MCP server boundary is the right call for ~340 LOC, the dual-backend dispatcher is structurally sound, and the stateless-server invariant (claim 1, claim 9) is honored — `index.js` imports neither `fs` nor `path` and performs zero file I/O. Claims 1, 2, 4, 5, 9, and 10 all hold up under code inspection.

However, three architectural defects gate approval. (1) `index.js` imports `zod` but `package.json` does not declare it — the build relies on a transitive dependency surfaced by `@modelcontextprotocol/sdk`. That is a contract breach: the runtime imports of a versioned module must appear in `dependencies`, otherwise an SDK minor-version bump or dependency tree change can break the server with no diff in this repo. (2) The Dockerfile `COPY index.js prompts.js ./` will fail in any clean checkout because `prompts.js` is gitignored — there is no documented step that produces it inside the Docker build context. CI / fresh clones cannot build the image as documented. (3) The min-observations guard (claim 3) is bypassable on empty input because `"".trim().split("\n")` yields `[""]` (length 1), which is less than 10 and rejects cleanly — but `"\n".repeat(10).trim().split("\n")` yields a length-9 array, rejecting input that the docs would consider valid. More importantly, the guard counts lines, not observations — a malformed log with one huge line counts as 1; ten blank lines count as 10. The contract in the README ("requires at least 10 lines in the observations log") is satisfied literally but not in spirit.

Beyond these, two warnings worth addressing: `package.json` version (1.0.0) and description ("via Ollama") drift from `index.js` (server version 2.0.0, dual-backend), and the `chat()` dispatcher silently ignores `options.temperature` on the Claude backend (Ollama honors it; Claude path doesn't even pass it). That last one violates claim 2 — backends are not actually equivalent under non-default options.

## Findings

### F-ARCH-001 — `zod` is an undeclared runtime dependency

**Severity**: critical
**Scenario**: `index.js:23` does `import { z } from "zod";`, but `package.json` declares only three dependencies: `@anthropic-ai/sdk`, `@modelcontextprotocol/sdk`, `supergateway`. `zod` resolves today only because `@modelcontextprotocol/sdk` pulls it in transitively (`package-lock.json:31` lists it under the SDK's `peerDependencies`/`dependencies`). The MCP SDK's `peerDependencies` declares `"zod": "^3.25.0 || ^4.0.0"`, which means it expects the consumer to provide it. If the SDK ever ships without a hoisted `zod` in `node_modules` (npm 7+ hoisting is heuristic, not contractual), or a future SDK release drops `zod` as a transitive, this server fails to start with `ERR_MODULE_NOT_FOUND`. This affects claim 5 indirectly — the import is module-top-level, so it executes BEFORE any backend dispatch, meaning even an Ollama-only deployment with a clean `npm ci` could fail to load. The AGENTS.md "Conventions" section says "Zod for tool schemas" as a convention, which makes the omission more glaring, not less.
**Suggested defense**: Add `"zod": "^3.25.0 || ^4.0.0"` to `package.json` `dependencies` and run `npm install` to refresh the lockfile. Match the version range the MCP SDK declares as its peer.

### F-ARCH-002 — Dockerfile depends on a gitignored file that nothing produces in the build context

**Severity**: critical
**Scenario**: `Dockerfile:6` reads `COPY index.js prompts.js ./`. `prompts.js` is gitignored (`.gitignore:2`) and AGENTS.md `Gotchas` line 61 explicitly states it must be produced by `cp prompts.template.js prompts.js`. The README's "Running as HTTP endpoint (Docker)" section (README.md:113–123) says only `docker build -t session-essence .` — no precondition that the operator first copy the template. This means: (a) a clean clone cannot build the Docker image without an undocumented step; (b) CI cannot build the image at all without manufacturing a fake `prompts.js`; (c) the symlinked path mentioned in AGENTS.md ("mcp-server-stack/servers/session-essence") only works for one operator. This violates claim 8 ("personalization is explicit, not auto-magic") in a different direction — explicit, yes, but the Docker contract assumes magic. It also creates traceability drift between README.md's documented build flow and the actual buildable shape of the repo.
**Suggested defense**: Either (a) have the Dockerfile copy `prompts.template.js` and run `cp prompts.template.js prompts.js` inside the image with a clearly-documented "operator must mount/override `prompts.js` for personalization" semantic, or (b) document the precondition in README.md "Running as HTTP endpoint" — `cp prompts.template.js prompts.js && docker build …`. Option (a) is the architecturally cleaner one: the Docker image should be self-buildable from a clean clone, and personalization should be a runtime concern (env var or volume mount), not a build-time mutation of a tracked file.

### F-ARCH-003 — `chat()` dispatcher does not pass `options.temperature` to Claude backend; backends are not equivalent under non-default options

**Severity**: warning
**Scenario**: `index.js:83–92` (`ollamaChat`) reads `options.temperature ?? 0.6` and forwards it. `index.js:57–64` (`claudeChat`) reads only `options.model` and `options.maxTokens` — `temperature` is silently dropped. The `chat()` dispatcher at `index.js:126–139` passes the same `options` object to whichever backend; nothing in its contract warns that `temperature` is honored on one path and ignored on the other. This violates claim 2 — operators should be able to switch backends without resetting their portrait, but the **synthesis behavior** is not equivalent because temperature defaults differ (Anthropic SDK default ~1.0; Ollama explicit 0.6). This also undercuts the design.md claim "Same prompts, same output shape" — the prompts match, but the inference parameters don't. Today no call site passes `temperature` so this is latent, but the function contract is misleading and the first caller to add a temperature override will hit an asymmetric behavior bug that's invisible in the dispatcher.
**Suggested defense**: Either (a) wire `temperature` through `claudeChat` (`client.messages.create({ ..., temperature: options.temperature ?? 0.6 })`) so the two backends honor the same option set, or (b) document in a comment above `chat()` that `temperature` is Ollama-only and reject the option in the Claude path. Option (a) is the architecturally right call because it preserves the symmetric-dispatcher invariant the rest of the code assumes.

### F-ARCH-004 — Min-observations guard counts lines, not observations; the contract surface is wrong

**Severity**: warning
**Scenario**: `index.js:178–188` rejects when `observations.trim().split("\n").length < 10`. This counts newlines, not parsed JSONL entries. Three concrete failure modes: (a) ten blank lines pass the guard; (b) a single very long observation with embedded `\n` (e.g., a tool response that wasn't escaped during hook write — JSON.stringify SHOULD escape, but the README's `jq -c` output is "one JSON object per line" only if downstream writes don't introduce literal newlines) counts as multiple lines; (c) the guard fires AFTER `.trim()`, so leading/trailing whitespace is collapsed correctly, but `""` still hits `.split("\n")` and yields `[""]` (length 1) — the empty-observation case rejects with `Insufficient observations (1 < 10 minimum)` which is technically accurate but semantically wrong (zero is not one). This affects claim 3 — the guard fires before any backend call, which is correct — but the contract advertised in README.md:301 ("requires at least 10 lines in the observations log") is exposed at the wrong abstraction level. Observations are JSONL records, not lines.
**Suggested defense**: Parse the input as JSONL and count successfully-parsed records: `const records = observations.split("\n").map(s => s.trim()).filter(Boolean).filter(s => { try { JSON.parse(s); return true; } catch { return false; } }); if (records.length < 10) …`. This makes the guard match the docs' intent ("10 observations") and gracefully drops malformed lines instead of counting them. As a bonus, it gives the synthesis pass a cleaner upstream guarantee.

### F-ARCH-005 — `package.json` version + description drift from `index.js` server identity

**Severity**: warning
**Scenario**: `package.json:3` says `"version": "1.0.0"` and `:4` says `"description": "…via Ollama — captures personality, …"`. `index.js:147` registers the MCP server as `version: "2.0.0"`. The feature commit `9b1436e` is "dual-backend synthesis (Claude API default, Ollama fallback)" — the description is stuck in the pre-dual-backend era. Plan claim 2 ("dual backends produce equivalent portraits") rests on the package presenting itself as dual-backend; the npm metadata says Ollama-only. Operators using `npm info` or `package.json` to understand the project will get the wrong picture; the MCP server's reported version (2.0.0) and the npm package version (1.0.0) diverge, which breaks the standard "package.json is the source of truth for version" expectation.
**Suggested defense**: Bump `package.json` `version` to `"2.0.0"` to match the MCP server registration, and rewrite the `description` to reflect dual-backend reality (e.g., "MCP server that synthesizes AI session portraits via dual-observer analysis — Claude API or Ollama backend"). Pick one source of truth for version and have the other reference it (e.g., import `version` from `package.json` into `index.js`).

### F-ARCH-006 — `analyze_portrait` system prompt is inline; breaks the "prompts are versioned in prompts.template.js" convention

**Severity**: suggestion
**Scenario**: `index.js:312–315` contains an inline system prompt: `"You are comparing two session portraits…"`. AGENTS.md `Conventions` line 56 says: "Prompts are versioned in `prompts.template.js`. When you change a prompt, change it in the template." This inline prompt is a prompt by every definition the AGENTS.md cares about, yet it lives in `index.js` and is not personalizable. It's also the only place a prompt change would NOT route through the template → operator-copy flow. Low-impact today (it's stable and doesn't reference the operator's name), but it's a foreseeable place a future maintainer would forget to update if the prompt convention evolves. This is a plan-traceability gap — the convention exists, the code mostly honors it, but this one tool drifts.
**Suggested defense**: Export `COMPARE_SYSTEM` (or similar) from `prompts.template.js` (and therefore `prompts.js`), then import and reference it from `analyze_portrait`. Keeps every system prompt in one file, makes the convention enforcement uniform.

### F-ARCH-007 — Dockerfile pins `SYNTHESIS_BACKEND=claude` but the image is intended to be runtime-switchable; defaults leak the backend assumption

**Severity**: suggestion
**Scenario**: Dockerfile lines 8–11 bake in `ENV SYNTHESIS_BACKEND=claude`, `OLLAMA_HOST=…`, `OLLAMA_MODEL=qwq:32b`, `CLAUDE_MODEL=claude-haiku-4-5-20251001`. AGENTS.md describes this as "Sets `SYNTHESIS_BACKEND=claude` by default. Operators who want Ollama override at `docker run` time." That's a fine choice, but the Dockerfile ALSO bakes in Claude-specific model + Ollama-specific host as ENV. Mixing both in the image isn't wrong (env precedence works), but it's a small smell: the image ships with both backends' defaults pre-populated, then the operator picks one at runtime. The dual-backend reality is reflected, but the README's `docker run` examples (README.md:115–125) hard-code different env subsets per backend, suggesting the docs and the Dockerfile aren't quite agreeing on what the image's contract is. Plan claim 2 (backend equivalence) and the audit question "does the Dockerfile reflect the dual-backend reality, or does it leak backend-specific defaults?" — answer: mostly yes, but the leak is `SYNTHESIS_BACKEND=claude` being baked into the image rather than being a runtime default with no ENV value at all. If the image were truly backend-neutral, it would leave `SYNTHESIS_BACKEND` unset and rely on `index.js:31`'s default.
**Suggested defense**: Drop the `ENV SYNTHESIS_BACKEND=claude` line — `index.js:31` already defaults to `"claude"` when the var is unset, so this line is duplicative. Then the Dockerfile defaults remaining (`OLLAMA_HOST`, `OLLAMA_MODEL`, `CLAUDE_MODEL`, `PORT`) are pure value defaults rather than backend selection. The image becomes truly backend-neutral.

### F-ARCH-008 — `format_observation` schema does not validate or enforce note length; downstream truncation in hooks is the only bound

**Severity**: suggestion
**Scenario**: `index.js:280–285` accepts `note: z.string()` with no `.max()` constraint. The hook contract elsewhere (README.md:189) truncates user prompts to `[0:2000]` and tool responses to `[0:500]`/`[0:300]`. The MCP tool's `format_observation` is a manual-entry path used by the calling Claude when it notices something the hooks miss — there's no length cap, so a very long `note` rides into the observations log unbounded and eventually flows back through `synthesize_essence`'s `lines.slice(-200)` truncation as a single oversized line. Not a security issue, not a correctness issue, but an asymmetric contract: hooks bound their entries; the manual tool doesn't. Plan claim 4 ("observation truncation is bounded") holds at the synthesis boundary but the upstream input shape is unbounded.
**Suggested defense**: Add `.max(4000)` (or whatever line cap matches the spirit of the hook truncation) to the `note` schema in `format_observation`. Document the cap as part of the tool contract.

## Cross-reviewer notes

- **For reviewer-sec**: The `analyze_portrait` tool takes two strings (`old_portrait`, `new_portrait`) and forwards them into a Claude system-prompted call with no sanitization. If portraits can contain prompt-injection payloads (and they can — the portrait is a second-person "You are…" document by design, which is structurally identical to a system prompt), then `analyze_portrait` is a prompt-injection delivery vector that doesn't even need observations. Worth a hard look on the security lens.
- **For reviewer-sec**: `index.js` does not log `ANTHROPIC_API_KEY` directly, but `console.error("Fatal:", err)` at line 340 will print the full error object. The Anthropic SDK has been known to include request headers in some error variants — worth checking whether a 401 or network error could surface the Authorization header (with the API key) into stderr where supergateway might capture it. Likely fine, but a sec-lens problem to verify.
- **For reviewer-sec**: The `examples/synthesize-essence.sh` uses `read -r -d '' PROMPT << 'SYNTHESIS_PROMPT'` with a quoted heredoc — that's the right defense against shell expansion in the prompt body itself. But the script also invokes `claude -p --permission-mode bypassPermissions` with `--allowedTools "Read Write Bash"`. That means a hostile observation that survives the prompt-injection defenses can issue arbitrary `Bash` commands in the operator's home directory. Security lens should rate this.
- **For reviewer-perf**: `chat()` runs three sequential awaits in `synthesize_essence` (psychologist, sociologist, merge). Phases 1 and 2 are independent — they could run in parallel via `Promise.all`. Sequential is architecturally fine (simpler error handling, clearer logs), but the perf lens may want to call out the latency-doubling cost of the sequential pattern, especially on the Claude backend where each call is a network round-trip.
- **For reviewer-perf**: `OLLAMA_TIMEOUT=600000` (10 minutes) bounds the Ollama path, but the Claude path has no timeout at all — `client.messages.create` uses the SDK default (which IIRC is none / very long). A hung Claude API connection blocks the entire MCP tool indefinitely. Perf-lens question on whether that's deliberate.
